Skill 不是让我更快的东西

发布于 2026-05-28 评论 生活 

尝试用 skills 写的文章
现在的思考，笔记不需要再花额外的时间去维护一个知识库
和 Agent 对话的过程就是产生内容的过程

原文

AI Coding 一年实战：从效率幻觉到能力进化
这篇是腾讯技术那边的一篇长文。标题和开头都很有冲击力：METR 的实验显示，资深开发者用了 AI 辅助编码以后，客观上慢了 19%，但主观上却觉得自己快了 20%。我一开始点进去，其实只是想快速学习一下，看看现在大厂架构师怎么理解 AI Coding 这件事，结果读完以后发现它讲的不是「AI 到底有没有用」这么简单的问题。

文章的主线可以压成一句话：AI 把 coding 加速了，但 coding 往往不是整个软件交付流程的瓶颈。

这句话听起来很普通，但它后面接了很多具体数据。比如 Faros AI 看到高 AI 采纳团队的个人指标变好，任务数、PR 数都涨了，可组织层面的 DORA 指标没有明显改善。再往下看，原因也不玄学：PR 体积变大，review 时间变长。也就是说，上游写得更快，下游审得更慢，最后系统吞吐并没有真的变好。

作者用约束理论解释这个现象，我觉得这个类比挺好记。工厂里如果瓶颈在热处理炉，你把旁边一个非瓶颈工位升级成高速机器人，整个工厂并不会因此交付更快，只会多出一堆在制品。AI 现在很像那个高速机器人。它很擅长把代码写出来，但需求理解、兼容性分析、Code Review、测试验证这些环节没有跟上，最后快出来的东西只是换了个地方排队。

后面文章开始拆 AI Coding 框架。OpenSpec、Superpowers、BMAD、Spec Kit 这些名字看起来很多，容易让人焦虑，好像不选一个正确框架就会落后。但作者的判断是，它们底层都在做同一件事：把软件开发流程里可复用的 prompt 和操作流程沉淀成 Skill。名字不同，包装不同，但本质都是在管理上下文和工作流。

这里最值得我记下来的，是 Spec、Rule、Skill 的区别。

Spec 是业务知识，告诉 AI「现在是什么、要做什么」；Rule 是工程规范，告诉 AI「哪些事情不能乱来」；Skill 是工作流，告诉 AI「这件事具体怎么做」。它们不是同义词。混在一起以后，Rule 会越写越长，变成一份什么都塞的杂货铺；Skill 如果只写成线性 SOP，又会丢掉 explore、brainstorm 这种真正需要发散思考的能力。

作者说 Skill 遵循性更好，不是因为它名字更高级，而是因为加载机制更合理。Rule 常驻在上下文头部，内容一多就有大量噪声；Skill 是按需加载，靠近生成位置，和当前任务相关性更高。换成我比较好理解的说法就是：Rule 像全局变量，Skill 像按需 import 的模块。全局变量多了以后，谁都不知道当前函数到底依赖哪一个。

再往后是验证闭环。作者用了丰田「安灯绳」的比喻：机器可以跑很快，但发现质量问题时必须能停下来。对应到 AI Coding，就是 Lint、Unit Test、E2E、Code Review 这些机制要接进工作流里。AI 写完不是结束，写完以后能不能自动验证、失败以后能不能修复，才决定它是不是一条可信的生产线。

最后是并发。文章承认，走完整流程以后，单任务可能确实变慢。但如果验证闭环足够可靠，人就可以同时调度多个 AI session。收益不来自「一个任务更快」，而来自「多个任务可以并行推进」。当然并发也不能无限开，否则 review 队列又会爆掉，重新回到第一章说的下游堆积。

所以这篇文章最后落到的观点，不是教人怎么追求更快，而是提醒人不要只追求更快。AI Coding 真正的红利不一定是效率提升，而是能力进化：去做以前做不到、懒得做、做不展开的事情。

我读完以后，最想接着写的其实就是这里。因为我这段时间用 skill 的体感，恰好也不是「我变快了」，而是「有些原来启动不了的事情，现在终于能启动了」。

模糊

读到这里的时候，我其实有一点被戳到。

因为我平时用 AI，尤其是用各种 skill 的时候，最强烈的体感并不是「好快」。当然它有时候确实快，尤其是生成代码、生成文字、改格式的时候，屏幕上的内容一行一行往外冒，很容易让人误以为事情已经完成了一半。但后来我慢慢发现，真正让我离不开它的原因，可能不是快，而是它能把我脑子里那些很模糊的东西先拽出来。

我脑子里经常有一些想法。说没有也不是，说清楚又很难。它们像一团湿毛巾，里面确实有水，但不拧就只是沉沉地搭在那里。比如我知道自己想写一篇关于 skill 的博客，知道它和 AI Coding 的效率悖论有关，也知道它和我最近玩的 ICPC、diary、雅思训练有关系。但这些东西最开始只是几块浮在水面的木板，彼此之间没有桥。

以前遇到这种情况，我会很容易放弃。打开博客后台，新建文章，写两行，然后觉得不对。删掉。再写三行，又觉得太像公众号。再删掉。最后浏览器标签页就那么开着，像一个很安静的指责。

现在 skill 至少能帮我做一件事：先把桥搭出来。

这件事本身不神秘。它无非是把模糊的意图拆成结构，把散乱的材料放到不同的篮子里，再用一种相对稳定的方式提醒我：先说原文讲了什么，再说你自己为什么被它击中，再说三个具体 skill 分别改变了什么。听起来很普通。但很多时候，我缺的恰好就是这个普通的结构。

Diary

diary 这个 skill 对我来说，是最像镜子的一个。

它不一定告诉我新的观点，但它会把我已经有的情绪照得更清楚一点。比如我说「我觉得自己脑子里有想法但是很模糊」，它不会立刻把这句话改成一个很漂亮的结论，而是会逼我回到更具体的东西：夜里的屏幕、还没写完的草稿、消息提示、空白的 Markdown 文件、删掉又恢复的句子。

这很重要。因为我原来写东西很容易直接跳到抽象层，讲焦虑，讲成长，讲选择，讲效率，讲能力进化。讲得都没错，但读起来像飘在半空中。diary skill 的好处是，它会让我先落地。先从桌子、键盘、宿舍、食堂、某个凌晨开始，再慢慢往里面走。

更直接一点说，它能帮我做到我想要的文字美。

这句话说出来有点不好意思，好像承认自己很在意修辞。但我确实在意。不是那种堆词的漂亮，而是希望一段文字读起来有呼吸，有一点迟疑，有一点回头看自己的动作。以前我想写这种东西，经常写成矫情。现在它能帮我把那种矫情压低一点，让句子看起来没那么急着证明自己。

当然，风险也在这里。

文字变美以后，人会误以为自己想清楚了。段落顺了，隐喻成立了，结尾也有一点余味，我就很容易产生一种虚假的满足感，好像这件事已经被我消化了。但其实也可能只是被包装好了。混乱还是混乱，只是现在有了比较好看的外壳。

所以 diary skill 不能替代思考。它更像是帮我把房间里的灯打开。灯亮了以后，灰尘还在，乱堆的衣服也还在。只是我终于能看见它们在哪里。

ICPC

ICPC skill 对我来说，情绪就复杂很多。

我经常会想，如果大一刚开始刷题的时候就有这种工具，会不会好很多。那时候最痛苦的不是写不出代码，而是看不懂题解。题面像一堵墙，题解像另一堵墙，中间没有路。最可怕的是，你甚至不知道自己该问什么。

很多题解的写法都很吝啬。一句「显然可以二分」，一句「考虑树形 DP」，一句「维护一个单调队列即可」，读起来像高手之间交换暗号。对已经会的人来说当然清楚，对不会的人来说，那个「显然」就是最不显然的地方。

我以前在这种地方花过很多时间。盯着一篇题解看半小时，看懂了每个汉字，但不知道它为什么能从 A 跳到 B。然后去搜别人的博客，博客也只是在另一个角度重复同一个结论。再去问人，又怕自己的问题太基础。最后就很容易产生一种不争气的念头：也许我就是不适合这个。

如果那时候有 ICPC skill，它最有价值的地方大概不是直接给我 AC 代码，而是把中间省略掉的台阶补回来。先问这题属于什么模型，再问为什么能这样转化，再拿一个小样例手推，再指出边界和反例，最后才写代码。它像是在一段很陡的坡上多钉几块木板，让我至少知道脚下一步该踩哪里。

这和原文里说的「能力进化」是对得上的。AI 不是让我更快地抄题解，而是让我有机会进入以前进不去的理解层。以前我卡在门外，连门把手在哪里都不知道。现在它至少能告诉我：你先别急着写代码，这里真正的问题是状态怎么定义。

不过这里也有另一层担心。

算法训练里有一部分能力，确实来自痛苦地卡住。卡住会逼人自己建模，自己试错，自己从错误里拆出规律。Skill 如果太顺手，可能会把这种笨拙的肌肉也一起省掉。最后我不是变强了，而是变成一个更会索要提示的人。

所以我理想里的 ICPC skill 应该更像教练，不像代打。先给方向，不给答案；再给关键观察，不给完整推导；实在不行再展开题解。它应该延迟满足，而不是立刻把糖塞到嘴里。这个分寸其实很难，但也正是 skill 设计里最有意思的地方。

雅思

雅思 skill 给我的感觉又不一样。

它不像 ICPC 那样有明确的 AC，也不像 diary 那样能立刻看到一段漂亮文字。雅思训练最大的痛苦是缺少正反馈。背单词、练口语、改作文、做阅读，这些事情都知道有用，但今天做完以后到底进步了多少，很难看见。

这种事情最消耗人。因为人不是机器，不能只靠长期收益活着。每天打开口语题库，对着空气讲两分钟，录音里听到自己的停顿和重复，心很容易凉半截。作文也是一样，一篇写完，好像完成了任务，但如果没有反馈，它就只是另一个被丢进文件夹里的文档。

雅思 skill 对我的意义，是把这种无正反馈的事情稍微游戏化了一点。

口语可以被拆成素材、故事线、part 3 追问、高分表达。作文可以被拆成审题、逻辑、词汇、语法、句子级修改。阅读错题不再只是「又错了」，而是可以拆成定位失败、同义替换没识别、True/False/Not Given 的判断标准混乱。

这些反馈不一定很宏大，但它们像经验条。哪怕只涨一点，也会让人觉得今天没有白练。对我这种很容易在长期任务里泄气的人来说，这种小反馈比鸡血有用得多。

不过游戏化也有它的问题。人会为了经验条而做任务，为了打卡而练习，为了看到「升级」而忽略真正的语言能力。雅思 skill 如果只让我沉迷在被评价、被鼓励、被总结里，那它也只是另一种更精致的拖延。

真正有用的地方，应该是它把我推回训练本身。让我多说一分钟，多改一句话，多看一次错题原因。反馈只是灯，不是路。

Skill

这样看下来，我玩的三个 skill 刚好对应了三个瓶颈。

diary 解决的是表达瓶颈。我不是没有想法，而是想法没有形状。

ICPC 解决的是理解瓶颈。我不是只缺答案，而是缺从题目走到题解的中间桥梁。

雅思解决的是反馈瓶颈。我不是不知道要练，而是练习本身太像一条没有灯的路。

这也解释了为什么我对「AI 到底有没有提效」这个问题越来越不敏感。它当然有时提效，有时不提效，有时甚至会让我慢下来。尤其是当我让它参与更完整的流程时，先分析、再规划、再执行、再验证，单看一个任务，确实比我直接莽过去慢。

但我现在越来越觉得，慢一点不一定是坏事。

如果 diary skill 让我把一个原本说不清的念头写成文章，这不是提速，而是让一件原本不会发生的事发生了。如果 ICPC skill 让我看懂一道原本会放弃的题，这不是提速，而是让我多走进一个知识区域。如果雅思 skill 让我在低反馈的训练里多坚持几天，这也不是提速，而是让一条快断掉的线又接上了一点。

原文说 AI Coding 的真正红利不在加速，而在能力进化。我现在愿意把这句话说得更小一点：skill 的价值不在于替我完成任务，而在于改变我和任务之间的距离。

有些任务原来离我太远，我看不清。它把中间放大。

有些任务原来太散，我抓不住。它给一个容器。

有些任务原来太闷，没有回声。它造一点反馈。

这大概就是我最近对 skill 的理解。它不是万能的，也不是某种新潮的外挂。用不好，它会让我更依赖，更虚假满足，更容易把漂亮输出误认为真实能力。但用得好，它可以成为一种脚手架，把那些原本卡在模糊、畏难、无反馈里的事情，变成可以开始迭代的事情。

至于脚手架搭起来以后，我到底会不会往上爬，这件事它也替不了我。

说到底，AI 也好，skill 也好，最后都只是把问题重新推回我面前：你到底想用省下来的力气做什么。

如果只是洗更多衣服，那可能也没什么意思。

这一篇就先写到这里吧。

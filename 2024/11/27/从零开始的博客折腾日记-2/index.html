一：Mkdocs 食用手册

关于文档的存放，前前后后换了好几个方案了。

  • 最开始的时候用的是 WordPress，后来发现功能太少，漏洞过多，就转战 hexo 了；

  • hexo 呢挺好的，但是 npm 不太好用，很多 bug 修的莫名其妙的，部分主题也长时间不维护了，我也挺喜欢用 typography 这个主题的，用来写写随笔挺好的；

  • 语雀，飞书，notion 也都不错，但是缺乏可定制性。

综上，再添加一个 Mkdocs 就可以满足我的需求啦，用过 mkdocs 后，我突然意识到所谓的 docs 的组织结构，我还蛮需要这种东西的，好多东西一旦尘封的时间长了，就会被大雪深埋～

二：一些思考

关于自己的技术栈？听了字节青训的老师还有师傅的建议后，我应该反思几个问题

  • 自己技术深度
  • 个人的优势

技术深度很好理解，我的技术水平可能在广度上还不错，但深度还没达到一个足够的水平，这是一个很致命的问题。个人优势是指在大环境下，前后端依然是一个不错的从业方向，前端和后端招聘的人数都差不多，但是卷后端的大佬好多，导致后端不好找到合适的工作，面试的时候面试官拷打你是想知道你哪里比较突出，在同样技术水平的人里面，面试官肯定会综合学历等等优先录取。

三：不需要解决的问题

​ 因为 typography 主题代码高亮无法正常显示，尝试去解决，发现如果我用 highlight.js 可以正常使用，但是用 prism 这个插件就没有成功加载，可能是别的 css 重新渲染掉了的原因？但是定位到那个 css 需要大量时间，我才体会到之前学到的有些问题根本不需要解决这观点，折腾这个对于当下的我来说并不是很划算的一个买卖，如果给的时间足够肯定是能解决的，现在嘛～能用就行，关注需求，关注自己技术栈的深度！
